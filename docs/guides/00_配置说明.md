# 配置说明

本文档描述 go-data-flow 的两种配置文件：**应用配置** 和 **Flow 配置**。

## 配置文件概览

| 配置类型 | 文件 | 用途 | 命令行参数 |
|---------|------|------|-----------|
| 应用配置 | `app.yaml` | 全局设置：日志、指标、并行模式 | `-a <file>` |
| Flow 配置 | `*.yaml` | 数据流定义：Source → Processor → Sink | `-c <file>` 或 `-d <dir>` |

---

## 1. 应用配置 (app.yaml)

应用配置用于设置全局运行参数。

### 完整示例

```yaml
# 应用名称
name: dataflow

# 日志配置
log:
  level: info
  format: text
  output: logs/dataflow.log
  prefix: "[dataflow]"
  max_size: 100
  max_backups: 5
  max_age: 30
  compress: true

# Prometheus 指标配置
metrics:
  enabled: false
  addr: ":9090"
  path: "/metrics"
  namespace: "dataflow"

# 并行运行模式
parallel: true

# 默认缓冲区大小
buffer_size: 100
```

### 配置项说明

#### 顶层配置

| 配置项 | 类型 | 默认值 | 说明 |
|-------|------|-------|------|
| `name` | string | `"dataflow"` | 应用名称，用于日志和监控标识 |
| `parallel` | bool | `true` | 是否并行运行多个 Flow（仅通过配置文件设置） |
| `buffer_size` | int | `100` | 默认通道缓冲区大小（可被 Flow 配置覆盖） |

#### log 日志配置

| 配置项 | 类型 | 默认值 | 说明 |
|-------|------|-------|------|
| `level` | string | `"info"` | 日志级别：`debug`、`info`、`warn`、`error` |
| `format` | string | `"text"` | 日志格式：`text`、`json` |
| `output` | string | `"stdout"` | 输出目标：`stdout`、`stderr`、或文件路径 |
| `prefix` | string | `"[dataflow]"` | 日志前缀 |
| `max_size` | int | `100` | 单个日志文件最大大小（MB），超过自动切割 |
| `max_backups` | int | `5` | 保留的旧日志文件最大数量 |
| `max_age` | int | `30` | 保留旧日志文件的最大天数 |
| `compress` | bool | `false` | 是否压缩旧日志文件（.gz 格式） |

> **注意**：`max_size`、`max_backups`、`max_age`、`compress` 仅当 `output` 为文件路径时生效。

#### metrics 指标配置

| 配置项 | 类型 | 默认值 | 说明 |
|-------|------|-------|------|
| `enabled` | bool | `false` | 是否启用 Prometheus 指标服务 |
| `addr` | string | `":9090"` | HTTP 监听地址 |
| `path` | string | `"/metrics"` | metrics 暴露路径 |
| `namespace` | string | `"dataflow"` | 指标命名空间前缀（如：`dataflow_records_in_total`） |

启用后，可通过 `http://localhost:9090/metrics` 访问 Prometheus 指标。

## 2. Flow 配置

Flow 配置定义数据处理管道：Source → Processors → Sink。

### 完整示例

```yaml
# Flow 名称（必填）
name: my-data-flow

# 通道缓冲区大小（可选，默认 100）
buffer_size: 100

# 数据源配置（必填）
source:
  name: source-generator-sequence
  config:
    count: 100
    interval_ms: 100

# 处理器链（可选，按顺序执行）
processors:
  - name: processor-condition-filter
    config:
      field: status
      op: eq
      value: 200
  - name: processor-transform-field
    config:
      add:
        processed: true
      mapping:
        old_field: new_field

# 输出目标配置（必填）
sink:
  name: sink-output-console
  config:
    format: json
    limit: 1000
```

### 配置项说明

#### 顶层配置

| 配置项 | 类型 | 默认值 | 必填 | 说明 |
|-------|------|-------|------|------|
| `name` | string | - | ✅ | Flow 名称，用于日志、监控和指标标签 |
| `buffer_size` | int | `100` | ❌ | 组件间通道的缓冲区大小 |
| `source` | ComponentSpec | - | ✅ | 数据源组件配置 |
| `processors` | []ComponentSpec | `[]` | ❌ | 处理器列表，按顺序执行 |
| `sink` | ComponentSpec | - | ✅ | 输出目标组件配置 |

#### ComponentSpec 组件规格

每个组件（source/processor/sink）都使用相同的配置结构：

```yaml
name: <组件名称>      # 必填：注册到框架的组件名
concurrency: 4        # 可选（仅 Processor）：覆盖并发数
config:               # 可选：组件特定配置
  key1: value1
  key2: value2
```

| 配置项 | 类型 | 必填 | 说明 |
|-------|------|------|------|
| `name` | string | ✅ | 组件名称，用于从注册表查找组件 |
| `concurrency` | int | ❌ | Processor 并发数覆盖。仅对声明支持并发的 Processor 生效。`1` 强制单线程，`N > 1` 使用 N 个 worker。不设置则使用组件默认值。对 Source 和 Sink 无效。建议根据 CPU 核心数和组件是否 I/O 密集来设置 |
| `config` | map | ❌ | 组件特定的配置参数，传递给 `Init` 方法 |

## 3. 命令行使用

### 基本命令

```bash
# 运行单个 Flow
./bin/dataflow -a ./conf/app.yaml -c ./conf/flows/flow.yaml

# 运行目录下所有 Flow（并行或顺序由 app.yaml 中的 parallel 配置决定）
./bin/dataflow -a ./conf/app.yaml -d ./conf/flows/

# 运行多个指定 Flow
./bin/dataflow -a ./conf/app.yaml -C ./conf/flows/flow1.yaml,./conf/flows/flow2.yaml

# 列出所有已注册组件
./bin/dataflow -l

# 显示版本
./bin/dataflow -v
```

### 命令行参数

| 参数 | 说明 |
|-----|------|
| `-a <file>` | 应用配置文件路径 |
| `-c <file>` | 单个 Flow 配置文件路径 |
| `-d <dir>` | Flow 配置文件目录 |
| `-C <files>` | 多个 Flow 配置文件，逗号分隔 |
| `-l` | 列出所有已注册组件 |
| `-v` | 显示版本信息 |

> **注意**：`parallel` 配置已移至 `app.yaml` 中设置，不再通过命令行参数控制。

### 目录扫描规则（`-d` 参数）

使用 `-d` 参数扫描配置目录时，遵循以下规则：

**识别的文件类型**

仅匹配 `*.yaml` 和 `*.yml` 两种扩展名，不会匹配 `*.json` 或其他格式。

**文件读取顺序**

分两轮扫描，分别按文件名字典序排列：

1. 先匹配 `*.yaml`，按文件名字典序排列
2. 再匹配 `*.yml`，按文件名字典序排列，追加到后面

即所有 `.yaml` 文件排在所有 `.yml` 文件前面，不是混合字典序。

示例，目录下有以下文件：

```
02-b.yaml
01-a.yml
03-c.yaml
00-d.yaml
```

实际读取顺序为：

```
00-d.yaml → 02-b.yaml → 03-c.yaml → 01-a.yml
```

> **提示**：建议统一使用 `.yaml` 扩展名，并用数字前缀（如 `01-xxx.yaml`）控制执行顺序。

**支持的格式对比**

| 方式 | 支持的格式 |
|------|-----------|
| `-d <dir>` 目录扫描 | 仅 `*.yaml`、`*.yml` |
| `-c <file>` / `-C <files>` | `.yaml`、`.yml`、`.json`，以及无扩展名文件（自动尝试 YAML → JSON 解析） |

## 4. 配置优先级

当同一配置项在多处定义时，优先级如下（从高到低）：

1. **Flow 配置** (`buffer_size`)
2. **应用配置** (`buffer_size`)

示例：
```yaml
# app.yaml
buffer_size: 50

# flow.yaml
buffer_size: 200  # 这个会生效
```

## 5. buffer_size 参数详解

`buffer_size` 控制组件之间 Go channel 的缓冲区大小，是影响数据流性能的关键参数。

### 工作原理

```
Source → [channel(buffer_size)] → Processor → [channel(buffer_size)] → Sink
           ↑ 缓冲区，可暂存 N 条记录
```

框架在组件之间创建带缓冲的 channel：

```go
// flow.go
channels[i] = make(chan T, f.config.BufferSize)
```

### 通道满时会发生什么

当缓冲区满了，**上游组件会阻塞等待**，直到下游消费数据腾出空间。这是 Go channel 的天然背压（Backpressure）机制：

```
Source 发送快 → 缓冲区满 → Source 阻塞等待 → 下游消费 → 缓冲区有空位 → Source 继续
```

**具体行为**：

| 场景 | 行为 |
|------|------|
| Source 发送过快 | Source 的 `Read()` 方法阻塞在 `out <- record` |
| Processor 处理慢 | 上游 Processor 或 Source 阻塞 |
| Sink 写入慢（如网络 I/O） | 整个管道逐级阻塞，形成背压 |

> **注意**：阻塞是同步的，不会丢失数据。当 `context` 取消时，阻塞的发送会解除。

### 不同大小的权衡

| 缓冲区大小 | 内存占用 | 吞吐量 | 背压响应 | 适用场景 |
|-----------|---------|--------|---------|---------|
| 小 (10-50) | 低 | 受限 | 快速传递压力 | 低延迟、流量平稳、内存敏感 |
| 中 (100-500) | 适中 | 良好 | 平衡 | 通用场景、开发调试 |
| 大 (1000-10000) | 高 | 高吞吐 | 延迟背压传递 | 高吞吐、流量突发、下游有延迟 |

### 场景推荐

| 场景 | 推荐值 | 说明 |
|------|-------|------|
| 开发调试 | `100`（默认） | 够用，便于观察阻塞问题 |
| 生产环境 | `1000-5000` | 应对正常流量波动 |
| 高吞吐/突发流量 | `10000+` | 吸收流量峰值，需确保内存充足 |
| 下游有网络 I/O（Kafka、HTTP） | `2000-5000` | 缓冲网络延迟带来的处理间隙 |
| 实时性要求高 | `10-100` | 快速传递背压，避免数据堆积 |

### 配置示例

```yaml
# 高吞吐场景
name: high-throughput-flow
buffer_size: 10000
source:
  name: source-kafka-consumer
  config:
    topic: events
processors:
  - name: processor-transform-field
sink:
  name: sink-elasticsearch

# 低延迟场景
name: low-latency-flow
buffer_size: 50
source:
  name: source-http
sink:
  name: sink-websocket
```

## 6. 配置文件格式

支持 YAML 和 JSON 两种格式：

```yaml
# YAML 格式（推荐）
name: my-flow
source:
  name: generator
  config:
    count: 100
```

```json
// JSON 格式
{
  "name": "my-flow",
  "source": {
    "name": "generator",
    "config": {
      "count": 100
    }
  }
}
```
